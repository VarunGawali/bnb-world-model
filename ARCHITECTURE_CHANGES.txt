================================================================================
BnB WORLD MODEL — ARCHITECTURE CHANGES
================================================================================

This document records every architectural change made to the model, explaining
what was changed, why, and what the expected impact is. File locations are
given relative to the repository root.

--------------------------------------------------------------------------------
CHANGE 1: SAGEConv → GATv2Conv  (bnb_wm/model/encoder.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    The two bipartite message-passing layers (constraints→variables and
    variables→constraints) previously used SAGEConv (Hamilton et al., 2017).
    They now use GATv2Conv (Brody et al., 2022) with n_heads=4 attention heads.

WHY
    SAGEConv aggregates neighbour embeddings by computing a simple mean (or
    max, depending on the aggregator). Every neighbouring constraint contributes
    equally to a variable's updated embedding, regardless of how much that
    constraint actually constrains the variable.

    In a MIP, constraints vary enormously in how "tight" they are at a given
    B&B node. A constraint that is near its RHS (almost binding) has far more
    branching relevance than one with large slack. Attention allows the GNN to
    learn which constraints to focus on for each variable, rather than treating
    them all equally.

    GATv2 specifically fixes a representational limitation in the original GAT
    (Veličković et al., 2018): GAT's attention scores can only express a
    monotone (linear) relationship between query and key, while GATv2 uses a
    joint transformation before the scoring linear layer, making it strictly
    more expressive.

TECHNICAL DETAILS
    - n_heads = 4; each head operates on hidden_dim // n_heads = 32 dimensions
    - concat=True so the output dim remains hidden_dim (same as before)
    - add_self_loops=False (bipartite graph; self-loops don't make sense)
    - edge_dim=3 passed so edge features are injected into the attention
      computation (see Change 2 below)

EXPECTED IMPACT
    Medium–High. The encoder is the backbone of all downstream predictions;
    attention-weighted aggregation should directly improve policy Top-1 accuracy
    and value head Spearman ρ.

CHANGE 2: Edge Features Added  (bnb_wm/model/encoder.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    The encoder now accepts an optional edge_attr tensor of shape [E, 3].
    When provided, each edge feature is injected into the GATv2 attention
    computation. The three edge features are:

        dim 0 : constraint coefficient  A_{ij}
        dim 1 : normalised coefficient  A_{ij} / (|RHS_i| + 1e-8)
        dim 2 : sign of coefficient     sign(A_{ij}) ∈ {-1, 0, +1}

    If edge_attr is not provided (e.g. during testing), it defaults to zeros
    so the model remains backward-compatible.

WHY
    The constraint matrix A is the core structure of a MIP. Two constraints
    with the same number of variables can have wildly different coefficients;
    those coefficients directly determine the LP relaxation and thus the
    branching importance of each variable. Without edge features, the GNN
    treats all edges identically and has no way to distinguish A_{ij} = 0.001
    from A_{ij} = 1000. Adding A_{ij} as an edge feature gives the GNN the raw
    information it needs to reason about constraint tightness.

    The normalised version A_{ij} / RHS_i measures the variable's relative
    contribution to the constraint's capacity, which is scale-invariant.
    The sign feature helps distinguish covering constraints (≥, positive
    coefficients) from other constraint types.

TECHNICAL DETAILS
    - Edge features are split by direction (c2v / v2c) alongside edge indices
    - GATv2Conv uses edge features in the attention score: the edge embedding
      is added to the key before the tanh + linear scoring step
    - The data generation pipeline (bnb_wm/data/generate.py) should populate
      edge_attr when constructing PyG Data objects

EXPECTED IMPACT
    High. Constraint coefficients are the most information-dense signal in a
    MIP; making them available to the GNN substantially increases the model's
    theoretical expressivity.

CHANGE 3: global_mean_pool → CrossAttentionPool  (bnb_wm/model/encoder.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    The graph-level embedding z was previously computed by
    global_mean_pool(h_vars, batch), which takes the unweighted average of
    all variable embeddings. It is now computed by CrossAttentionPool, a
    learned readout that uses a single trainable query vector to attend over
    all variable embeddings in a graph.

    The mechanism is identical to the [CLS] token in BERT:
        keys   = W_k(h_vars)          # [n_vars, H]
        values = W_v(h_vars)          # [n_vars, H]
        attn   = softmax(q · keys^T / sqrt(H))
        z      = attn · values         # [H]

    where q is the single learnable query vector.

WHY
    Mean pooling treats all variable nodes as equally important for the
    graph-level summary. In practice, the dual bound and leaf probability
    are dominated by a small subset of structurally critical variables
    (highly fractional, high objective coefficient, tightly constrained).

    CrossAttentionPool can learn to focus its query on those variables,
    producing a graph embedding that better summarises the "difficulty" of the
    current B&B node. This directly benefits the value head and integrality
    head, both of which consume z.

TECHNICAL DETAILS
    - query : nn.Parameter of shape [1, H] (one learnable token)
    - W_k, W_v : Linear(H, H, bias=False)
    - Softmax over the variable dimension (not batch dimension)
    - Loops over batch indices; can be replaced with scatter_softmax for
      GPU efficiency if batch sizes are large

EXPECTED IMPACT
    Medium. The value head Spearman ρ and integrality head accuracy are most
    likely to benefit. The policy head benefits indirectly via the Pointer
    Network (Change 4) which also uses z.

CHANGE 4: MLP PolicyHead → Pointer Network  (bnb_wm/model/heads.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    The policy head previously scored each variable independently:
        score_i = MLP(h_var_i)

    It now uses a Pointer Network (Vinyals et al., 2015) that scores each
    variable in the context of the global graph embedding z:

        query  = W_q(z_per_var)                       # global context
        key_i  = tanh(W_k(h_var_i) + W_z(z_per_var)) # local + global mix
        score_i = v^T (query_i ⊙ key_i) / sqrt(H)

    where z_per_var is z broadcast to each variable node in the same graph.

    The forward signature changed:
        old: forward(h_vars)
        new: forward(h_vars, z_per_var)

WHY
    Strong branching evaluates all candidate variables jointly by solving
    an LP for each and comparing the resulting bound improvements. The
    original independent MLP has no way to compare candidates to each other —
    it assigns an absolute score to each variable in isolation.

    The Pointer Network introduces a coupling: the global embedding z
    summarises the full B&B state, and each variable is scored relative to
    that global context. This is a soft form of "all candidates are considered
    simultaneously," which is much closer to how the expert (strong branching)
    actually works.

    The attention-weighted scoring also naturally handles variable-length
    candidate sets and is permutation-equivariant (same as the MLP version).

TECHNICAL DETAILS
    - W_q, W_k, W_z : Linear(H, H, bias=False)
    - v              : Linear(H, 1, bias=False)
    - z_per_var is computed in BnBWorldModel.forward() as z[batch.batch[var_mask]]
    - During Phase 1 training, the loss masking over the candidate action set
      is applied after this forward pass, same as before

EXPECTED IMPACT
    Medium. Policy Top-1 accuracy should improve because the head can express
    relative importance. The effect may be subtle on Set Cover (a structured
    problem) but should be more pronounced on harder, heterogeneous instances.

CHANGE 5: ValueHead with Fractional Variable Mean  (bnb_wm/model/heads.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    The value head previously computed:
        v = MLP(z)    [z is graph-level embedding, H-dim]

    It now computes:
        frac_mean = mean(h_vars[fractional_variables])   # [H]
        v = MLP(concat(z, frac_mean))                    # input is 2H

    If no fractional mask is available, frac_mean falls back to z (safe).

    The forward signature changed:
        old: forward(z)
        new: forward(z, h_vars, batch_vec, frac_mask=None)

WHY
    The LP dual bound at a B&B node is determined by the fractional variables —
    which variables are not yet fixed to integer values and by how much they
    deviate from integrality. The global mean pool z captures the overall
    graph structure but may average away the fractional structure if most
    variables are already integer.

    Explicitly computing the mean embedding of fractional variables gives the
    value head a direct signal about the current fractional state, which is the
    most predictive feature for the dual bound. This is a cheap, targeted
    enrichment that adds no extra GNN computation.

TECHNICAL DETAILS
    - frac_mask : boolean tensor [total_vars], True = variable is fractional
    - The fractional mean is computed per graph using batch_vec
    - MLP depth increased slightly: Linear(2H, H) → ReLU → Linear(H, H//2)
      → ReLU → Linear(H//2, 1) to handle the richer 2H input

EXPECTED IMPACT
    Medium. Target: lift Spearman ρ from 0.682 toward 0.80+.

CHANGE 6: IntegralityHead with Auxiliary Scalars  (bnb_wm/model/heads.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    The integrality head previously computed:
        logit = MLP(z)   [bottleneck: H → H//2 → 1]

    It now computes:
        logit = MLP(concat(z, depth, n_frac))   [input: H + 2]

    where:
        depth  : integer depth of the current node in the B&B tree
        n_frac : number of fractional variables at this node

    Both scalars are normalised to float and concatenated as [batch, 1] each.
    If not provided, they default to zeros (backward-compatible).

    The forward signature changed:
        old: forward(z)
        new: forward(z, depth=None, n_frac=None)

WHY
    A leaf node is one where all variables are integer (n_frac == 0) or the
    node is pruned. Both depth and n_frac are among the strongest predictors
    of whether a node is a leaf:
        - n_frac: if n_frac == 0, the node IS a leaf by definition
        - depth:  leaf nodes cluster at large depths; shallow nodes almost
                  never are leaves

    The GNN embedding z cannot reliably infer these quantities. Depth is not
    encoded in any node feature and would require special positional encoding.
    n_frac requires counting across all variable nodes, which mean pooling
    destroys (it averages fractional values, losing the count). Adding them
    as explicit scalars is a principled and cheap fix.

TECHNICAL DETAILS
    - Scalars are float-cast and unsqueezed to [batch, 1] before concat
    - MLP input is H + 2 instead of H; all other dimensions unchanged
    - Training pipeline should provide these from the B&B node metadata
      stored in each .npz trajectory file

EXPECTED IMPACT
    Low–Medium. Integrality head accuracy should improve substantially for
    near-leaf nodes (high depth, low n_frac), which is exactly where correct
    prediction matters most for pruning decisions.

CHANGE 7: DynamicsGRU → DynamicsTransformer  (bnb_wm/model/dynamics.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    The dynamics model previously used a single GRUCell:
        z_{t+1}, h_{t+1} = GRU([z_t || a_t], h_t)

    It is now a causal Transformer decoder with n_layers=4 and n_heads=4:
        token_t = Linear([z_t || a_t])         # project to d_model
        x       = [token_0, ..., token_t]      # causal sequence
        z_{t+1} = Transformer(x)[-1]           # read last position

    The Transformer uses:
        - Causal (upper-triangular masked) multi-head self-attention
        - Pre-norm (LayerNorm before each sub-layer)
        - GELU activation in the FFN (4× expansion)
        - Learned positional embeddings (not sinusoidal)

    Two interfaces are provided:
        forward(z_seq, a_seq)  : parallel training over full trajectories
        step(z_t, a_t, past)   : single-step inference with explicit KV cache

WHY
    The GRU compresses the entire trajectory history into a fixed-size hidden
    vector h_t. By design, the GRU forgets information exponentially as the
    sequence grows: a branching decision made 20 nodes ago has very small
    influence on h_t. But in Branch-and-Bound, early decisions constrain the
    entire subtree — the variable fixed at depth 2 is active at every node
    below it, making long-range dependencies essential.

    The Transformer has full receptive field: every token can attend to every
    past token with equal weight if the attention scores warrant it. This allows
    the dynamics model to reason about the accumulated effect of the entire
    branching history, not just the last step.

    Additional advantages:
        (a) Multi-step lookahead: the same Transformer can predict z_{t+k} by
            unrolling forward k times, enabling planning without LP calls.
        (b) Parallel training: the full trajectory is processed in one forward
            pass with teacher forcing, which is much faster than GRU's
            sequential teacher-forcing (all T steps in O(1) Transformer call
            vs. T sequential GRU steps).
        (c) Scaling: Transformers scale predictably with depth and width;
            GRUs don't benefit from additional layers in the same way.

TECHNICAL DETAILS
    - d_model = hidden_dim (same as encoder, no bottleneck)
    - n_layers = 4, n_heads = 4, head_dim = 32
    - FFN expansion = 4× with GELU + dropout(0.1)
    - max_seq = 512 (causal mask is pre-allocated; trajectories are typically
      much shorter than 512 nodes)
    - Causal mask is registered as a buffer (not a parameter; moves with .to())
    - step() maintains a token buffer [B, t, H] passed in and returned each
      call; this avoids re-computing KV pairs from scratch at each step

EXPECTED IMPACT
    High. This is the largest single architectural improvement. The original
    GRU was trained in Phase 3 but never used at inference because it couldn't
    meaningfully capture trajectory context. The Transformer's richer context
    encoding makes it practical to use dynamics_step() during branching
    (1-step lookahead) or during tree search (multi-step latent planning).

================================================================================
SECTION 2: BRANCH-AND-CUT PIPELINE
================================================================================

The changes in this section complete the full pipeline: every model component
is now wired into training and inference, and a Python-owned Branch-and-Cut
solver is implemented from scratch.

--------------------------------------------------------------------------------
CHANGE 10: CuttingPlaneHead  (bnb_wm/model/heads.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    A new prediction head was added: CuttingPlaneHead.

    Architecture: identical to PolicyHead (Pointer Network) but operating on
    cuts instead of variables.

        cut_emb_k = ReLU(W_in · cut_feat_k)         # project 6-dim → H
        key_k     = tanh(W_k · cut_emb_k + W_z · z) # local + global mix
        score_k   = v^T (key_k) / sqrt(H)            # scalar score per cut

    Input  : cut_feats [n_cuts, 6], z [H]  (single node embedding)
    Output : scores    [n_cuts]

    Cut features (6-dim per candidate cut):
        0  violation     : rhs - lhs@x*         (how much LP solution violates)
        1  efficacy      : violation / ||lhs||   (normalised bite)
        2  density       : nnz(lhs) / n_vars     (sparsity)
        3  parallelism   : cos(lhs, c)           (alignment with objective)
        4  obj_cutoff    : estimated bound improvement
        5  support_frac  : fraction of cut support that is fractional

WHY
    The choice of which cuts to add in branch-and-cut critically affects
    solver performance. Adding too many cuts slows LP solving; adding weak
    cuts wastes time without tightening the bound. Classical heuristics
    (maximum violation, minimum density) treat each cut independently.

    The Pointer Network allows the head to score all candidate cuts jointly
    given the current B&B node state z. This means the model can learn
    context-dependent preferences: preferring dense cuts at the root (where
    they propagate to many nodes) and sparse cuts at deep nodes (where LP
    resolve cost matters more).

    The head is trained to imitate SCIP's cut selection decisions (Phase 5):
    given the same candidate pool, predict which cuts SCIP would choose and
    which of those actually improved the LP bound. This is a supervised
    imitation signal that is cheap to generate alongside trajectory data.

TECHNICAL DETAILS
    - cut_feat_dim = 6 (configurable via BnBWorldModel constructor)
    - W_in : Linear(6, H)
    - W_k, W_z : Linear(H, H, bias=False)
    - v : Linear(H, 1, bias=False)
    - At inference: sigmoid(score) >= threshold (default 0.3) selects cuts
    - The head processes one node at a time (z is a single [H] vector, not
      batched) because the number of candidate cuts varies per node

EXPECTED IMPACT
    High (at solver level). The CuttingPlaneHead is the key component that
    makes the system a genuine branch-and-cut solver rather than just a
    branching policy. It controls the tradeoff between cut tightness and LP
    solve overhead at every node.

CHANGE 11: cutting_plane_loss  (bnb_wm/training/losses.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    A new loss function was added: cutting_plane_loss().

    Weighted binary cross-entropy:
        loss = BCE_with_logits(scores, labels, pos_weight)

    where labels[k] = 1 if cut k was selected by SCIP and improved the LP
    bound by > threshold, 0 otherwise.

WHY
    Most candidate cuts are not useful (high class imbalance). pos_weight
    corrects for this by upweighting positive examples (good cuts) in the
    loss. This is the same technique used for the integrality head (leaf
    nodes are also rare).

    The label definition — "selected by SCIP AND improved the bound" — is
    important. Cuts that SCIP selected but didn't tighten the bound are not
    labeled positive; this teaches the model to prefer cuts with actual
    impact rather than mimicking SCIP's selection heuristic blindly.

TECHNICAL DETAILS
    - pos_weight = n_negative_cuts / n_positive_cuts per batch
    - Computed from the training data's cut_labels distribution
    - Identical interface to integrality_loss for consistency

CHANGE 12: Phase 5 training — train_cuts()  (bnb_wm/training/trainer.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    A new training method train_cuts() was added to the Trainer class.

    Phase 5 freezes the encoder and trains only the CuttingPlaneHead:
        1. Encode node → z  (encoder frozen, no gradient)
        2. model.cut_scores(cut_feats, z[b]) → scores per cut
        3. cutting_plane_loss(scores, cut_labels, pos_weight)
        4. Backprop through CuttingPlaneHead only

    Loader format: same (pyg_batch, metas) as other phases, with each
    meta containing:
        cut_features : Tensor [n_cuts, 6]
        cut_labels   : Tensor [n_cuts]  binary

    After Phase 5 completes, all parameters are unfrozen so Phase 4
    joint fine-tuning can include the cut head.

WHY
    Training the cut head separately (before joint fine-tuning) follows
    the same staged curriculum as the other heads: learn one prediction
    target well before coupling all losses together. Joint training from
    scratch with 4 different loss terms (policy + value + integrality +
    cut) would require careful hyperparameter tuning of 4 loss weights;
    staged training avoids this.

TECHNICAL DETAILS
    - Graphs without any candidate cuts are skipped silently
    - Validation: per-cut BCE loss (not accuracy, since imbalance is high)
    - Best checkpoint saved when val_loss improves
    - All parameters unfrozen at end of phase for subsequent joint training
    - Training phase ordering: 1 → 2 → 3 → 5 → 4 (cuts before joint)

CHANGE 13: Trainer call-site fixes  (bnb_wm/training/trainer.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    Three call sites in the existing trainer were broken by the architecture
    changes (new head signatures). They were fixed:

    (a) _run_value_batch:
        Old: model.value_pred(z)
        New: model.encode(pyg_batch) → h_vars, z
             var_mask, batch_vec computed from pyg_batch.node_type
             frac_mask derived from LP value feature (index 9 of x[var_mask])
             model.value_pred(z, h_vars, batch_vec, frac_mask)

    (b) train_joint — value_pred:
        Same fix as above (h_vars + batch_vec + frac_mask required)

    (c) train_joint — integrality_logit:
        Old: model.integrality_logit(z)
        New: depth extracted from meta["depth"] (default 0)
             n_frac extracted from meta["n_frac"] (default 0)
             model.integrality_logit(z, depth, n_frac)

    (d) train_joint — policy scores:
        Old: scores, z = model(pyg_batch) — then scores used directly
        New: h_vars, z = model.encode(pyg_batch)
             z_per_var = z[pyg_batch.batch[var_mask]]
             scores = model.policy(h_vars, z_per_var)
             (This matches the Pointer Network's actual forward signature)

EXPECTED IMPACT
    These are correctness fixes. Without them, training phases 2 and 4
    would raise TypeError at runtime. No model capacity change.

CHANGE 14: Phase 3 training — train_dynamics()  (bnb_wm/training/trainer.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    The previously missing train_dynamics() method was implemented.

    Phase 3 freezes the encoder and trains the DynamicsTransformer:
        1. Loader yields (z_seq, a_seq, z_next_seq) pre-computed tensors
        2. model.dynamics_forward(z_seq, a_seq) → z_pred [B, T, H]
        3. dynamics_loss(z_pred, z_next_seq) = MSE + 0.1 * cosine
        4. Backprop through DynamicsTransformer only

    The SequenceDataset (to be implemented in bnb_wm/data/) is responsible
    for pre-computing z and action embeddings by running the frozen encoder
    on all graphs in each trajectory.

    The parallel forward pass (full sequence in one call) replaces the
    sequential GRU teacher-forcing that was implicit in the original design.
    Training a T-step trajectory now costs 1 Transformer forward pass
    instead of T sequential GRU steps.

CHANGE 15: BnBSolver — full branch-and-cut implementation
           (bnb_wm/solver/bnb_solver.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    All NotImplementedError methods were implemented. New dataclass CutData
    was added. The solver now runs a complete branch-and-cut loop.

    NODE REPRESENTATION
        Node.inherited_cuts : list[CutData]
            Globally-valid cuts propagated from ancestors. This list grows
            monotonically as we descend the tree (child = parent_cuts + new_cuts).
            Children of the same parent share the same cut list.

    LP SOLVING: _solve_lp()
        Uses scipy.optimize.linprog with method='highs' (HiGHS backend, same
        solver that SCIP/Gurobi use internally). Constraints are converted
        from Ax >= b to -Ax <= -b for scipy's A_ub convention. Inherited cuts
        are stacked as additional rows.

        Dual variables are extracted from result.ineqlin.marginals and used
        for node encoding (reduced costs, constraint activity).

    NODE ENCODING: _encode_node()
        Builds a 19-dim variable feature matrix and 5-dim constraint feature
        matrix matching the Ecole NodeBipartite observation format that the
        model was trained on, then encodes with the GNN.

        Variable features computed from LP solution:
            LP value, fractional part, basis status (at-lb/basic/at-ub),
            normalised objective, normalised reduced cost, cut count.

        Constraint features:
            Activity, relative saturation, dual value, normalised RHS,
            is-tight indicator.

        Edge features: A_{ij}, A_{ij}/b_i, sign(A_{ij}).

    CUT GENERATION: _generate_cg_cuts()
        Generates pairwise Chvátal-Gomory intersection cuts:
            For each pair (i1, i2) of tight constraints:
                inter = {j : A_{i1,j}=1 AND A_{i2,j}=1}
                cut  :  ∑_{j ∈ inter} x_j  >=  1

        These cuts are GLOBALLY VALID because they represent a CG round
        of (row_i1 + row_i2) / 2 with integer coefficients on the
        intersection support. They remain valid when variable bounds are
        tightened by branching.

        Enumeration is bounded:
            - Only tight constraints (activity ∈ [1.0, 1.4]) are paired
            - At most 30 tight constraints → C(30,2)=435 pairs maximum
            - Only violated cuts (intersection LP sum < 1 - eps) are kept
            - Duplicate supports are filtered
            - Top 50 by violation magnitude are returned as candidates

    CUT SELECTION: _select_cuts_neural()
        Computes 6-dim cut features, calls model.cut_scores(), applies
        sigmoid threshold and top-k cap, returns selected CutData list.

    BRANCHING: _select_branch_var()
        1. PolicyHead scores all variables
        2. Mask to fractional candidates (0 < x*_j < 1)
        3. DynamicsTransformer 1-step lookahead over top-k candidates:
               for each candidate a:
                   z_next = dynamics_step(z, h_vars[a], past_tokens)
                   v_next = value_pred(z_next, ...)
           Select candidate with highest predicted value
        4. This is the first time DynamicsTransformer is used at inference

    NODE PRIORITY
        IntegralityHead used to detect near-leaf nodes (leaf_prob > 0.7):
            - Near-leaf: skip cut generation (unnecessary computation)
        ValueHead score added to LP-based priority for node ordering:
            priority = -lp_obj + 0.01 * value_head_score
        This blends LP-based best-first with learned value estimates.

    BRANCH-AND-CUT (not cut-and-branch)
        child.inherited_cuts = parent.inherited_cuts + new_cuts
        Both children of a branch inherit all cuts from the parent plus
        cuts selected at the parent. This is correct because all generated
        cuts are globally valid and remain valid under any variable bound
        restriction.

TECHNICAL DETAILS
    - LP method: scipy linprog method='highs' (no extra install required)
    - Cut types: "cg" only (Chvátal-Gomory); Gomory fractional cuts are
      intentionally excluded (they are node-local and cannot propagate)
    - Max cuts per node: 10 (configurable in default.yaml)
    - Cut score threshold: 0.3 sigmoid value (configurable)
    - Lookahead k: 3 candidates (configurable)
    - Node ordering: max-heap by priority (higher = processed first)
    - Pruning: standard lb >= global_ub bound pruning
    - Termination: optimal gap < gap_tolerance, node_limit, or time_limit

CHANGE 16: Benchmark — full model at inference  (bnb_wm/evaluate/benchmark.py)
--------------------------------------------------------------------------------

WHAT CHANGED
    The GNN branching loop in run_macro_benchmark() was upgraded to use
    all model components at inference instead of just the policy head.

    Old loop (used only encoder + policy head):
        h_vars, _ = model.encode(batch)
        scores = model.policy_scores(h_vars)   # wrong signature
        best_action = scores[action_set].argmax()

    New loop (_gnn_pick_action):
        1. Encode → h_vars, z
        2. IntegralityHead → leaf_prob
           If leaf_prob > 0.8: skip lookahead, use policy scores directly
        3. PolicyHead (Pointer Network) → baseline scores for all candidates
        4. DynamicsTransformer 1-step lookahead over top-5 candidates:
               z_next = dynamics_step(z, h_vars[candidate], past_tokens)
               v_next = value_pred(z_next, ...)
           Select candidate with highest predicted next-state value
        5. Advance token buffer: past_tokens updated with chosen action
           (the buffer persists across all branching decisions in one instance,
            giving the Transformer full trajectory context)

    _format_obs() was updated to build edge_attr [E, 3] from
    obs.edge_features.values (constraint coefficients from Ecole).

    policy_scores() call was fixed to use the new 3-argument signature:
        model.policy_scores(h_vars, z, var_batch)

EXPECTED IMPACT
    High. The benchmark previously used only 2 of 5 model components. Now
    all 5 components contribute to inference. The dynamics lookahead is the
    most significant addition: instead of greedily picking the highest-scored
    variable, the model simulates one step ahead in latent space and picks
    the variable whose predicted future is best.
